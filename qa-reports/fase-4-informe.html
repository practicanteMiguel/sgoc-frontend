Bugs Corregidos
1
Flash de redireccion a login con sesion activa (race condition de hidratacion)
src/app/page.tsx
Severidad: Alta Tipo: Race condition · UX

La pagina raiz (/) leia isAuthenticated del store Zustand directamente sin esperar la hidratacion desde localStorage. En la primera renderizacion, Zustand devuelve su estado inicial (isAuthenticated: false) antes de leer el valor persistido. El useEffect disparaba inmediatamente y redirigfa a /auth/login. Un frame despues, la hidratacion terminaba, isAuthenticated se actualizaba a true y el efecto volvia a correr redirigiendo a /dashboard/dashboard. El usuario veia un flash de la pantalla de login aunque su sesion estuviera vigente. 

// Antes -- useEffect corre con isAuthenticated=false antes de la hidratacion
export default function HomePage() {
  const router = useRouter();
  const { isAuthenticated } = useAuthStore();

  useEffect(() => {
    if (isAuthenticated) {
      router.replace('/dashboard/dashboard');
    } else {
      router.replace('/auth/login');  // dispara en t=0 con false
    }
  }, [isAuthenticated, router]);
}

// Despues -- espera a que Zustand termine de leer localStorage
export default function HomePage() {
  const router = useRouter();
  const { isAuthenticated } = useAuthStore();
  const hasHydrated = useHasHydrated();

  useEffect(() => {
    if (!hasHydrated) return;   // bloquea hasta hidratacion completa
    if (isAuthenticated) {
      router.replace('/dashboard/dashboard');
    } else {
      router.replace('/auth/login');
    }
  }, [hasHydrated, isAuthenticated, router]);
}

Modal de confirmacion de contrasena se cerraba antes de que la mutacion completara
src/components/auth/change-password.tsx — funcion onConfirm()
Severidad: Alta Tipo: Error handling · UX

onConfirm() llamaba a setShowConfirmModal(false) de forma sincrona antes de ejecutar changePassword.mutate(). Si la API retornaba un error, el modal ya habia desaparecido: el usuario veia un toast de error pero sin posibilidad de reintentar desde el mismo modal -- tenia que completar el formulario entero de nuevo. El modal ya mostraba un estado de carga con isPending, por lo que cerrarlo anticipadamente era innecesario. 

// Antes -- modal se cierra antes de conocer el resultado
const onConfirm = () => {
  if (!pendingData) return;
  setShowConfirmModal(false);   // cierra inmediatamente
  changePassword.mutate({ ... });
};

// Despues -- modal permanece abierto; en exito el hook redirige a login
// y el modal desaparece con la pagina; en error queda abierto para reintentar
const onConfirm = () => {
  if (!pendingData) return;
  changePassword.mutate({ ... });
  // el boton muestra Loader2 mientras isPending=true
  // en exito: router.replace('/auth/login') desmonta el modal
  // en error: toast + modal sigue abierto, usuario puede reintentar
};

Expiracion de sesion redirige a login sin guardar la URL actual
src/lib/axios.ts — interceptor response (catch refreshError) · src/hooks/auth/use-auth.ts — useLogin onSuccess
Severidad: Alta Tipo: Navegacion · UX

Cuando el refresh token expiraba, el interceptor de Axios usaba window.location.href = "/auth/login" descartando la URL en la que estaba el usuario. Tras re-autenticarse, el usuario llegaba siempre al dashboard raiz aunque estuviera en medio de un flujo (por ejemplo, en la pantalla de solicitudes con filtros aplicados). El fix captura la URL actual como parametro returnUrl en el redirect, y el hook useLogin lo lee al completar el login exitoso. 

// axios.ts -- Antes
if (typeof window !== "undefined") {
  window.location.href = "/auth/login";
}

// axios.ts -- Despues: preserva la URL actual como returnUrl
if (typeof window !== "undefined") {
  const returnUrl = encodeURIComponent(
    window.location.pathname + window.location.search
  );
  window.location.href = `/auth/login?returnUrl=${returnUrl}`;
}

// use-auth.ts useLogin onSuccess -- Antes
router.replace('/dashboard/dashboard');

// use-auth.ts useLogin onSuccess -- Despues: redirige a la pagina original si existe
const returnUrl = new URLSearchParams(window.location.search).get('returnUrl');
router.replace(returnUrl ?? '/dashboard/dashboard');
4
Sin error boundaries -- crash en auth o dashboard produce pantalla en blanco
src/app/auth/error.tsx (creado) · src/app/dashboard/error.tsx (creado)
Severidad: Alta Tipo: Resiliencia · Error handling

Las rutas /auth/* y /dashboard/* no tenian archivo error.tsx. En Next.js 16, un error de renderizado no capturado en un segmento sin boundary hace que la pagina entera muestre una pantalla en blanco (o el error generico de Next.js en produccion). Un crash en el formulario de login o en cualquier modulo del dashboard dejaba al usuario sin opcion de recuperacion. Se crearon dos error boundaries que muestran el mensaje del error y ofrecen un boton de reintento. 

// src/app/auth/error.tsx -- creado
'use client';
import { useEffect } from 'react';
import { useRouter } from 'next/navigation';

export default function AuthError({ error, reset }) {
  const router = useRouter();
  useEffect(() => { console.error('[auth]', error); }, [error]);
  return (
    <div className="min-h-screen flex items-center justify-center">
      <p style={{ color: 'var(--color-danger)' }}>{error.message}</p>
      <button onClick={reset}>Reintentar</button>
      <button onClick={() => router.replace('/auth/login')}>Ir al login</button>
    </div>
  );
}

// src/app/dashboard/error.tsx -- creado (misma estructura, sin boton de login)
5
Logout no cancelaba requests HTTP en vuelo
src/hooks/auth/use-auth.ts — useLogout
Severidad: Alta Tipo: Gestion de estado · Requests

El hook useLogout llamaba a qc.clear() para limpiar el cache de TanStack Query, pero esto no cancela las peticiones HTTP que ya estaban en vuelo. Si un usuario cerraba sesion mientras una consulta de datos estaba pendiente, esa request podia resolverse despues del logout y generar errores de estado inconsistente o mostrar datos de la sesion anterior. Se agrego qc.cancelQueries() antes de qc.clear() tanto en onSuccess como en onError para enviar la senal de cancelacion a todas las queries activas. 

// Antes -- clear() limpia el cache pero no cancela requests activos
onSuccess: () => {
  qc.clear();
  clearAuth();
  router.replace('/auth/login');
},
onError: () => {
  clearAuth();
  router.replace('/auth/login');
},

// Despues -- cancelQueries() envia AbortSignal a todas las queries en vuelo
onSuccess: () => {
  void qc.cancelQueries();
  qc.clear();
  clearAuth();
  router.replace('/auth/login');
},
onError: () => {
  void qc.cancelQueries();
  clearAuth();
  router.replace('/auth/login');
},
