next.config.ts - IP de produccion hardcodeada como fallback del proxy
next.config.ts : 3
Impacto medio Infraestructura Configuracion

El proxy de Next.js usa NEXT_PUBLIC_API_URL para redirigir todas las llamadas /proxy/:path* al backend. Si la variable de entorno no esta definida en un nuevo entorno de despliegue, el fallback apuntaba silenciosamente a la IP interna 10.10.1.181:3001. Esto causa que en cualquier entorno nuevo sin el .env configurado correctamente, todas las llamadas de API fallen sin ningun error obvio de configuracion. La correccion cambia el fallback a cadena vacia, haciendo que el fallo sea inmediato y explicito. 

// ANTES - next.config.ts:3
const BACKEND_URL = process.env.NEXT_PUBLIC_API_URL ?? 'http://10.10.1.181:3001';

// DESPUES - falla explicito si la variable no esta configurada
const BACKEND_URL = process.env.NEXT_PUBLIC_API_URL ?? '';

header.tsx - Boton de notificaciones (campana) sin aria-label
src/components/layout/header.tsx : 107-116
Accesibilidad WCAG 2.1 AA Layout global

El boton de la campana de notificaciones no tiene aria-label ni title. Los lectores de pantalla (NVDA, VoiceOver) lo anuncian como "button" sin descripcion alguna, imposibilitando su uso a usuarios con discapacidad visual. Es un control siempre visible en el header global de la aplicacion, afecta todas las paginas del dashboard.

header.tsx - Boton de cambio de tema con title pero sin aria-label
src/components/layout/header.tsx : 96
Accesibilidad WCAG 2.1 AA Layout global

El boton de tema (sol/luna) tiene el atributo title con la etiqueta correcta pero no tiene aria-label. El atributo title genera un tooltip visual al hacer hover, pero no es anunciado de forma consistente por todos los lectores de pantalla. aria-label es el mecanismo semantico correcto para controles sin texto visible, y es requerido por WCAG 2.1 criterio 4.1.2 (Name, Role, Value). 

// ANTES - header.tsx:96
          title={theme === "dark" ? "Modo claro" : "Modo oscuro"}
        >

// DESPUES - aria-label agregado con el mismo valor dinamico que title
          title={theme === "dark" ? "Modo claro" : "Modo oscuro"}
          aria-label={theme === "dark" ? "Modo claro" : "Modo oscuro"}
        >

Sin error boundary raiz - rutas publicas con token sin cobertura de error
src/app/error.tsx (archivo creado)
Impacto medio Error handling 7 rutas publicas afectadas

El proyecto tenia error.tsx en cada sub-ruta del dashboard (/dashboard/error.tsx, /dashboard/consumables/error.tsx, etc.), pero carecia de un error.tsx raiz en src/app/. Las siguientes rutas publicas no tenian ninguna error boundary en su arbol de componentes: /vault/[token], /via-vault/[token], /dotaciones/[token], /dotaciones/autorizador/[token], /consumables/llenado/[id], /consumables/solicitudes/[id], /consumables/compras/[token]. Un error no capturado en estas rutas mostraba la pagina de error generica de Next.js. Se crea src/app/error.tsx como fallback global para todas las rutas sin boundary propio. 

// CREADO - src/app/error.tsx
'use client';

import { useEffect } from 'react';

export default function GlobalError({
  error,
  reset,
}: {
  error: Error & { digest?: string };
  reset: () => void;
}) {
  useEffect(() => {
    console.error('[global]', error);
  }, [error]);

  return (
    <div className="min-h-screen flex items-center justify-center p-8 bg-white">
      <div className="text-center max-w-sm">
        <p className="text-sm font-mono mb-1 text-red-600">Error inesperado</p>
        <p className="text-xs mb-6 text-gray-500">
          {error.message ?? 'Ocurrio un problema. Por favor intenta de nuevo.'}
        </p>
        <button onClick={reset} className="px-4 py-2 rounded-lg text-sm font-medium bg-blue-600 text-white">
          Reintentar
        </button>
      </div>
    </div>
  );
}

Falsos Positivos Documentados (4)
# Lo que parecia un problema Archivo Por que es correcto
1 console.error en todos los archivos error.tsx src/app/**/error.tsx (10 archivos) Patron intencional de Next.js App Router. Los error boundaries llaman console.error para que los errores de produccion aparezcan en los logs del servidor y herramientas de monitoreo. Es el comportamiento esperado y documentado por Next.js.
2 Sin framework de pruebas (Jest/Vitest/Playwright) package.json Es una deuda tecnica documentada, no una regresion introducida durante este ciclo QA. No existian tests antes de comenzar la Fase 1. Su ausencia no genera bugs en el codigo existente; es una mejora futura independiente de este ciclo.
3 Sin archivo .env.example documentando variables de entorno Raiz del proyecto Gap de documentacion que no afecta funcionalidad. Las variables requeridas (NEXT_PUBLIC_API_URL, NEXT_PUBLIC_WS_URL) estan referenciadas en el codigo. No constituye un bug de produccion.
4 IP real del backend en .env.local .env.local Archivo de entorno local de desarrollo. No se versiona (esta en .gitignore). La IP en .env.local es la configuracion correcta para el entorno de desarrollo del equipo. El bug corregido fue el fallback en next.config.ts, no el valor en .env.local.
