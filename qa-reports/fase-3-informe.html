Bug Critico Corregido
1
Estado dual: axios.ts manipulaba localStorage directamente sin actualizar Zustand
src/lib/axios.ts — funciones clearStoredAuth() y updateStoredAccessToken()
Severidad: Alta Tipo: Inconsistencia de estado de sesion

Las funciones clearStoredAuth() y updateStoredAccessToken() mutaban localStorage directamente, bypassando el store Zustand en memoria. Cuando el refresh de token fallaba, clearStoredAuth() vaciaba los tokens en disco pero isAuthenticated seguia siendo true en el store hasta que la pagina recargaba completamente. Durante esa ventana, cualquier hook o guard que leyera el store veia tokens nulos pero sesion activa.

El mismo problema en sentido inverso: al refrescar el token con exito, updateStoredAccessToken() actualizaba localStorage pero el store en memoria conservaba el token viejo. Cualquier codigo que leyera useAuthStore().accessToken directamente recibia el token expirado. 

// Antes - escritura directa a localStorage (bypassea Zustand en memoria)
function updateStoredAccessToken(token: string) {
  const parsed = JSON.parse(localStorage.getItem('auth-storage')!)
  parsed.state.accessToken = token
  localStorage.setItem('auth-storage', JSON.stringify(parsed))
}
function clearStoredAuth() {
  const parsed = JSON.parse(localStorage.getItem('auth-storage')!)
  parsed.state.accessToken = null
  parsed.state.refreshToken = null
  parsed.state.user = null
  parsed.state.isAuthenticated = false  // solo en disco, no en memoria
  localStorage.setItem('auth-storage', JSON.stringify(parsed))
}

// En el interceptor 401:
updateStoredAccessToken(newToken)   // store en memoria tiene token viejo
clearStoredAuth()                   // isAuthenticated sigue true en memoria

// Despues - fuente unica de verdad via Zustand (persist sincroniza a localStorage automaticamente)
import { getAuthState } from '@/src/stores/auth.store'

// En el interceptor 401 - refresh exitoso:
getAuthState().setAccessToken(newToken)   // actualiza memoria + localStorage

// En el interceptor 401 - refresh fallido:
getAuthState().clearAuth()                // isAuthenticated: false en memoria + localStorage

La funcion getStoredTokens() se conserva para el interceptor REQUEST porque lee directamente de localStorage para cubrir la ventana de arranque anterior a la rehidratacion de Zustand. Ese es el unico caso donde leer el storage directamente es correcto.

Riesgos Documentados — Deuda Tecnica de Seguridad
2
Refresh token y access token almacenados en localStorage (accesibles via XSS)
src/stores/auth.store.ts — clave: auth-storage
Severidad: Media Tipo: Almacenamiento inseguro de tokens Requiere cambio en backend

Ambos tokens se persisten en localStorage a traves del middleware persist de Zustand. localStorage es accesible via JavaScript desde cualquier script que corra en el mismo origen. Un ataque XSS exitoso podria leer los tokens y exfiltrarlos.

Riesgo actual:
  • Access token y refresh token en localStorage
  • Cualquier script XSS puede leerlos con localStorage.getItem('auth-storage')
  • El refresh token tiene vida util larga, su robo permite sesiones persistentes
Mitigacion recomendada:
  • Emitir el refresh token como cookie httpOnly; Secure; SameSite=Strict desde NestJS
  • El access token puede quedar en memoria (sin persistencia)
  • El interceptor de refresh llama al endpoint sin enviar el token manualmente (la cookie viaja sola)

Estado: riesgo acotado. Se verifico que el proyecto no usa dangerouslySetInnerHTML en ninguna parte (Fase 1) y React escapa automaticamente el contenido de las notificaciones y mensajes del servidor. La superficie XSS es baja. Se documenta como deuda tecnica dependiente del backend.

3
Sin middleware.ts — proteccion de rutas 100% client-side
src/components/auth/auth-guard.tsx — no existe src/middleware.ts
Severidad: Baja Tipo: Ausencia de proteccion server-side Depende del riesgo 2

La proteccion de rutas se basa en AuthGuard (client-side) que verifica el store Zustand despues de rehidratacion. No existe un middleware.ts de Next.js que valide la sesion en el Edge Runtime antes de servir la pagina.

Un usuario que manipule localStorage podria cargar el HTML inicial de una ruta protegida. Sin embargo, no obtendria ningun dato real porque todas las llamadas a la API requieren un Bearer token valido en el backend. El resultado seria un shell vacio con errores de carga.

Estado: riesgo aceptado. Implementar middleware.ts requiere validar el JWT en el Edge Runtime de Next.js sin acceso al store Zustand, lo que implica mover el access token a una cookie legible por el servidor. Esta mejora es dependiente del riesgo anterior.
