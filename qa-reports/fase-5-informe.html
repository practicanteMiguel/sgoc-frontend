useUpdateFacturas no invalida el cache de requisiciones
src/hooks/consumables/use-informe.ts · linea 16
MEDIO TanStack Query Cache consistency
useUpdateFacturas es el hook para la actualizacion masiva de facturas desde el modulo de compras. Al ejecutarse con exito solo invalidaba ['informe'], pero no ['requisiciones']. El problema: si un usuario tiene abierta la vista de detalle de una requisicion mientras otra persona guarda facturas en bloque, el detalle sigue mostrando los valores anteriores hasta que recarga manualmente. Contraste: useGuardarFacturas en use-requisiciones.ts (que actualiza facturas de una sola RQ) si invalida ambas keys. Ademas se agrego el toast de exito que faltaba. 
// ANTES
onSuccess: () => qc.invalidateQueries({ queryKey: ['informe'] }),
onError:   () => toast.error('Error al guardar facturas'),

// DESPUES
onSuccess: () => {
  qc.invalidateQueries({ queryKey: ['informe'] })
  qc.invalidateQueries({ queryKey: ['requisiciones'] })
  toast.success('Facturas actualizadas')
},
onError: () => toast.error('Error al guardar facturas'),

useListUsersSelect cachea resultado antes de que Zustand hidrate
src/hooks/users/use-users.ts · linea 106
MEDIO Zustand hydration TanStack Query key
El hook filtra al usuario actual con u.id !== me?.id, donde me viene de useAuthStore(). El bug: si el hook se ejecuta antes de que Zustand hidrate desde localStorage, me es undefined. En ese estado u.id !== undefined es siempre true para todos los usuarios, incluyendo al usuario actual. Con staleTime: 2 * 60 * 1000, ese resultado incorrecto queda cacheado bajo la key ['users', 'list-simple'] por 2 minutos. Aunque Zustand hidrate despues, la key no cambia y TanStack Query no refetch. La correccion agrega me?.id a la query key: cuando el usuario se vuelve disponible tras la hidratacion, la key cambia y se dispara un nuevo fetch con el filtro correcto. 
// ANTES — me?.id no esta en la key; el resultado stale persiste 2 minutos
queryKey: ['users', 'list-simple'],
queryFn:  () =>
  api.get(...).then((r) => r.data.data.filter((u) => u.id !== me?.id)),

// DESPUES — key cambia con me?.id; refetch automatico tras hidratacion
queryKey: ['users', 'list-simple', me?.id],
queryFn:  () =>
  api.get(...).then((r) => r.data.data.filter((u) => u.id !== me?.id)),

Sin error boundaries por modulo del dashboard
src/app/dashboard/*/error.tsx · 8 archivos creados
BAJO Next.js error boundaries UX
El dashboard tenia 9 rutas de modulo (consumables, dashboard, equipment, monitoring, reports, tools, users, vehicles, settings) pero solo un error.tsx en el nivel raiz src/app/dashboard/error.tsx. Cuando cualquier modulo lanza una excepcion, el boundary raiz captura el error y muestra un mensaje generico de "Error inesperado" sin identificar que modulo fallo. Se crearon error.tsx individuales para los 8 modulos restantes (excluido dashboard/ que ya tiene cobertura del raiz), con mensaje especifico por modulo.

Hallazgo reportado Archivo Por que es falso positivo
EncargadoFirmaModal — estado no se resetea al cerrar requisiciones-tab.tsx El modal se renderiza con showFirmaModal && <EncargadoFirmaModal>. Al cerrarse, React desmonta el componente por completo. Cada apertura es un mount fresco con estado inicial limpio.
startDraw / draw — closures stale en useCallback requisiciones-tab.tsx:109-126 getPos() solo captura canvasRef.current (ref estable, no estado) y las coordenadas del evento. No hay estado de React capturado que pueda volverse stale.
RevisionSolicitudModal — dep array [solicitud?.id] incompleto requisiciones-tab.tsx:309 Intencional: el comentario eslint-disable explicita que solo se reinicializa cantidades cuando cambia la solicitud, no en cada refetch en background. Evita resetear ediciones del usuario durante una revalidacion.
recepcion-modal.tsx — useMemo con eslint-disable en deps recepcion-modal.tsx:43-62 cantidades SI esta en el array de dependencias. El eslint-disable suprime la advertencia sobre getRec (funcion inline no memoizada), pero los datos que getRec usa (cantidades) si estan en los deps. El memo recalcula correctamente.
CreateInsumoModal — form no se resetea tras exito insumo-modal.tsx:83 onSuccess: onClose cierra el modal, lo que desmonta el componente. react-hook-form parte de defaultValues en cada mount. No hay estado residual.
useDotacionSpaceInfo — crash al parsear JSON en error use-dotaciones.ts:41 if (!r.ok) throw new Error('not found') ocurre antes de llamar r.json(). Solo se parsea JSON cuando r.ok === true. No hay ruta de codigo que crashee.
useFirmarAutorizador — race condition en FormData use-dotaciones.ts:81-83 FormData.append() es sincrono. Los tres append() se ejecutan secuencialmente en el mismo tick. No existe gap asincrono entre ellos.
