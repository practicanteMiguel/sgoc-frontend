useProfile - staleTime: 0 dispara refetch en cada montaje del modulo de monitoreo
src/hooks/settings/use-profile.ts : 13
Refetch innecesario Impacto de red Modulo Monitoreo

Con staleTime: 0, TanStack Query considera el cache inmediatamente obsoleto. Cada vez que el componente que usa useProfile se monta, se dispara una llamada a /users/profile. El hook es usado en voice-log-list.tsx (lista de transcripciones, siempre visible al navegar al modulo de monitoreo) y en profile-tab.tsx. El perfil contiene nombre, cargo y campo del usuario - datos que casi nunca cambian en una sesion. La mutacion useUpdateProfile ya llama a qc.invalidateQueries(['profile']) en su onSuccess, por lo que el refetch continuo es completamente redundante: cualquier actualizacion real del usuario ya invalidaria el cache inmediatamente.

Con staleTime: 5 * 60 * 1000 (5 minutos), el perfil se sirve desde cache durante la sesion normal de trabajo y se refresca automaticamente cuando el usuario lo actualiza. 

// ANTES - use-profile.ts:13
    return useQuery({
        queryKey: ['profile', user?.id],
        queryFn: () => api.get('/users/profile').then((r) => r.data),
        enabled: isAuthenticated,
        staleTime: 0,   // refetch en CADA montaje y window focus
    });

// DESPUES - datos de perfil son casi estaticos; mutacion invalida al actualizar
    return useQuery({
        queryKey: ['profile', user?.id],
        queryFn: () => api.get('/users/profile').then((r) => r.data),
        enabled: isAuthenticated,
        staleTime: 5 * 60 * 1000,
    });

useLogVault - refetchInterval: 15000 genera 4 llamadas por minuto para imagenes de bitacora
src/hooks/activities/use-vault.ts : 10
Polling excesivo 4 calls/min por bitacora abierta Modulo Bitacoras

useLogVault descarga la lista de imagenes asociadas a una bitacora de actividades. Con refetchInterval: 15000, si un supervisor mantiene una bitacora abierta por 1 hora, el browser realiza 240 llamadas a /logbook/{id}/vault solo para verificar si hay nuevas imagenes. El hook de carga useLogbook ya invoca qc.invalidateQueries({ queryKey: ['vault', logId] }) en su onSuccess, cubriendo las subidas del usuario actual de forma inmediata. El caso de subidas de otros usuarios en colaboracion de campo no requiere actualizacion en menos de 1 minuto - 60 segundos es suficiente para una experiencia colaborativa aceptable sin saturar el servidor. 

// ANTES - use-vault.ts:10
export function useLogVault(logId: string | null) {
  return useQuery({
    queryKey: ['vault', logId],
    queryFn: () => api.get<VaultImage[]>(`/logbook/${logId}/vault`).then((r) => r.data),
    enabled: !!logId,
    refetchInterval: 15000,   // 4 API calls/minuto por bitacora abierta
  })
}

// DESPUES - invalidacion por mutacion cubre subidas propias; 60s es suficiente para colaboracion
export function useLogVault(logId: string | null) {
  return useQuery({
    queryKey: ['vault', logId],
    queryFn: () => api.get<VaultImage[]>(`/logbook/${logId}/vault`).then((r) => r.data),
    enabled: !!logId,

  })
}

useViaVault - refetchInterval: 20000 genera 3 llamadas por minuto para grupos de captura
src/hooks/vias/use-via-vault.ts : 10
Polling excesivo 3 calls/min por via abierta Modulo Vias

useViaVault descarga los grupos de captura (fotos de verificacion de via) de un log de via especifico. A diferencia de useLogVault, no existe ninguna invalidacion por mutacion en el codebase para ['via-vault'] (verificado por grep global), por lo que el polling es el unico mecanismo para ver subidas de otros usuarios. Sin embargo, 20 segundos sigue siendo demasiado agresivo para imagenes que se capturan puntualmente durante inspecciones de campo. Aumentar a 60 segundos mantiene la visibilidad colaborativa con un tercio del trafico de red. 

// ANTES - use-via-vault.ts:10
export function useViaVault(logId: string | null) {
  return useQuery({
    queryKey: ['via-vault', logId],
    queryFn: () => api.get<ViaCaptureGroup[]>(`/via-logs/${logId}/captures`).then((r) => r.data),
    enabled: !!logId,
    refetchInterval: 20000,   // 3 API calls/minuto, sin invalidacion por mutacion
  })
}

// DESPUES

# Lo que parecia un problema Archivo Por que es correcto
1 Posible memory leak: useEffect(() => stopTimer, []) voice-recorder.tsx:30 Patron correcto de React. stopTimer es retornado como referencia de funcion cleanup - no llamado. React lo invoca en el unmount del componente, limpiando el setInterval. No hay leak.
2 useEmployees sin guard enabled cuando field_id podria ser nulo src/hooks/reports/use-employees.ts Todos los call sites que pasan field_id usan valores ya validados (ej. deliverable.field.id, field.id). Los calls de dashboard sin field_id son intencionales: vistas administrativas que requieren todos los empleados.
3 useNotifications + useUnreadCount ambos con refetchOnWindowFocus: true, generando 2 llamadas API por enfoque use-notifications.ts:89, 102 Intencional: son endpoints distintos (lista paginada vs. contador). El staleTime: 60 * 1000 limita los refetches a maximo 1 vez por minuto por enfoque, lo que es razonable para notificaciones en tiempo de aplicacion.
4 usePushStatus con staleTime: 0 en use-notifications.ts:40 use-notifications.ts:40 Solo se usa en la pagina de configuracion, no en el layout global. El impacto es una unica llamada async a la Service Worker API por enfoque de ventana - negligible en frecuencia e impacto. No justifica correccion.
5 useViaLog (20s) y useRequisicion (30s) con polling activado en vistas de detalle use-via-logs.ts:24, use-requisiciones.ts:34 Polling intencional para edicion colaborativa: varios usuarios pueden modificar el mismo via-log o requisicion simultaneamente. Los intervalos (20s y 30s) son aceptables para el caso de uso de campo. Se diferencian de los bugs porque los datos cambian frecuentemente por accion de otros usuarios.
